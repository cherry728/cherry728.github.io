从源码角度分析nextTick

了解nextTick之前，我们得掌握一点事件循环的知识。

事件循环

  1. 宏任务（task）：整体代码、setTimeout、setInterval、setImmediate、I/O、UI渲染（每种宏任务都有自己的一个队列）

  2. 微任务（jobs）：process.nextTick、promise回调、async/await、MutationObserver（每种微任务都有自己的一个队列）

Jake Archibald大神给出了经典动画演示了浏览器里面任务运行的原理

左边有缺口的环是执行任务队列，右边的环是执行浏览器渲染。浏览器的执行顺序是先执行任务队列，再去执行渲染步骤。

再看上面的例子，第一个例子中，while循环所在的代码实际上是在执行任务队列，所以浏览器进程一直在左边的环“旋转跳跃”不停歇，就永远到不了右边的环了，所以浏览器就无法渲染，新加入的任务也无法执行，所以浏览器卡死。第二个例子里面，浏览器每次只处理一个宏任务，所以每次执行完setTimeout，会“绕到”右边的环执行渲染工作。

总结：

  1. 一个循环过程：全局上下文进入函数调用栈 => 调用栈清空 => 执行所有微任务（执行微任务过程中，有新加入的微任务，就放入微任务队列执行，直到微任务队列清空才进行下一步） => 进入下一个循环

  2. 下一个循环：依然从宏任务task开始执行

然后看nextTick(flushSchedulerQueue)，这个flushSchedulerQueue是什么？

截取关键代码，flushSchedulerQueue去执行了watcher的run方法，这里的get实际上就是watcher做的render和update，也就是生成新的vnode树，然后更新到真实dom。

以上我们知道了把flushSchedulerQueue传入了nextTick方法，那nextTick又做了什么？（以下源码有部分删减）

nextTick主要是维护了一个队列，把传入的参数（就是上面我们说的watcher的更新操作）push到队列，然后执行timerFunc，这里是关键，其实timerFunc生成了一个微任务。

优先判断运行环境是否支持promise，支持就用promise.then，把更新操作加入微任务队列，不支持promise就判断是否支持MutationObserver，也是个微任务。再不支持MutationObserver，就判断是否支持setImmediate。最后降级到使用setTimeout来生成一个宏任务。

flushCallbacks其实是把nextTick维护的callbacks队列里面的方法遍历执行。

以上是系统级的数据更新，然后更新dom。如果用户使用了$nextTick方法，我们都知道$nextTick方法的回调函数里面，我们可以取得到最新dom。再来从源码角度分析一下。

这里的$nextTick，其实就是上面的nextTick方法，传入的回调函数加入callbacks队列，然后按照上面的步骤把回调加入微任务队列，执行完主代码，一次性执行微任务队列。
